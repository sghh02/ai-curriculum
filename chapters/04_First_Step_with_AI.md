# 第4章: AIと最初の一歩を決める

## 1. この章でやること
第3章で自分の方向性を整理しました。
この章では、AIと一緒に「最初に何を試すか」「何から学ぶか」を決めます。

いきなりコードを書き始めるのではなく、まず進め方を決めます。
ここで決めたことが、第5章で実際に試す内容になります。

## 2. AIにいきなり答えを頼まない
AIは便利ですが、最初にいきなり「〇〇を教えて」と答えを求めると、分からないまま進みやすくなります。

先に相談すべきことは次の4つです。

1. 最初は何だけやるか
2. 何から手をつければいいか
3. どこでつまずきそうか
4. 何を確認すれば「できた」と言えるか

## 3. この章で決めること
AIと一緒に、次を決めます。

- 最初に試す小さな課題
- 最初の3ステップ
- 先に学ぶとよいこと
- つまずきそうなポイント
- 試したあとに確かめること

## 4. AIへの聞き方
ここでのAIの使い方は、旧来の「検索」とは違います。
自分の整理結果を渡したうえで、具体的に進め方を相談します。

質問の例:

```text
私は未経験で、〇〇方向のエンジニアを目指しています。
第3章で次のように整理しました。

方向性: [ここに書く]
最初の「できた」の基準: [ここに書く]
今はやらないこと: [ここに書く]
分からないこと: [ここに書く]

この内容をもとに、
1. 最初にやること
2. 最初の3ステップ
3. 先に学ぶとよいこと
4. つまずきやすいポイント
5. 進んだか確かめる方法
を教えてください。
```

```text
この進め方で、もっと小さく始めるならどうすればいい？
```

**なぜこの質問が良い質問なのか:**
自分の整理結果を先に渡しているので、AIは一般論ではなく自分に合った回答を返せます。
さらに「もっと小さく」と聞き直すことで、実行可能な粒度まで落とせます。

## 5. 試す前の準備: Gitで記録を残せるようにする
第5章で小さく試すとき、「何が変わったか」「いつ戻れるか」が分かるようにしておくと安心です。
そのためにGitを最低限だけ使います。

### まず覚えること
最初は次の5つで十分です。

- `git status` — 今の状態を見る
- `git diff` — 何が変わったか見る
- `git add` — 変更を記録対象にする
- `git commit` — まとまりごとに記録する
- `git log --oneline` — 履歴を見る

### 使い方の流れ
1. 作業前に `git status` を見る
2. 変更したあとに `git diff` を見る
3. 問題なければ `git add` する
4. まとまりごとに `git commit` する
5. 最後に `git log --oneline` で履歴を見る

### この段階では深くやらないこと
- 複雑なブランチ運用
- rebaseの詳細
- 履歴のきれいな整理

今必要なのは、怖がらずに記録を残すことです。

### GitについてもAIに聞いてよい

- 「この変更は1回でコミットしてよい？」
- 「この変更内容に合うコミットメッセージを考えて」
- 「この差分で気をつける点を教えて」

## 6. 良い進め方の特徴
良い進め方には、次の特徴があります。

- 最初の作業が軽い
- 途中で進歩が見える
- 分からない点がはっきりしている
- 次に聞くべき質問が見えている

## 7. この章のゴール
この章が終わったら、次がそろっている状態を目指します。

- 最初に試す小さな課題が決まっている
- 最初の3ステップが書けている
- 先に学ぶことがリストになっている
- Gitの5つのコマンドを知っている
- 第5章で試す準備ができている

**【ワーク】**
第3章の整理結果をAIに見せて、「初心者向けの最初の一歩」を相談してください。
そのあと「もっと小さく始めるならどうする？」と聞いて、さらに1段階小さくしてください。

また、Gitの `status -> diff -> add -> commit -> log` を1回だけ試してみてください。

---
**【この章のまとめ】**
- AIには、まず答えではなく進め方を相談します。
- 自分の整理結果を渡すことで、AIの回答の質が上がります。
- Gitは「安心して試すための道具」として最低限だけ使います。
- 大事なのは、最初の一歩を実行可能な大きさまで小さくすることです。
