# コースガイド：AI時代の思考OSアップデート

このカリキュラムは、プログラミング言語やツールの前に行う「共通土台」です。  
目的は、AIの操作手順を覚えることではなく、**自分の思考を設計できる状態**を作ることです。

---

## このコースの理念

- AIは魔法の杖ではない。空気は読めないが超優秀な実行部隊である。
- AIは鏡である。曖昧な指示を出すと、曖昧な結果で返ってくる。
- 評価されるのは一発回答の精度ではなく、問いの質・分解力・再設計力である。

## 到達目標

このコース終了時に、次を満たしていることを目指します。

1. 曖昧な依頼を、5W1Hと変数に分解できる
2. 未知の要素を調査し、選定理由を説明できる
3. 処理順序と条件分岐を日本語で設計できる
4. 仕様変更が来ても、差し替え箇所を即座に特定できる

## 全体構成（2週間集中）

| 週 | フェーズ | 学習内容 | 目安時間 |
|---|---|---|---|
| 1週目 | PHASE 1 | 構造化解体（分解） | 180分 |
| 1週目 | PHASE 2 | 調査の技術（仮説と検証） | 240分 |
| 2週目 | PHASE 3 | ロジック設計（擬似コード化） | 240分 |
| 2週目 | PHASE 4 | ストレスリハーサル（破壊と再構築） | 240分 |
| 2週目 | 最終ゲート | Git接続判定 | 120分 |

## 共通お題

全フェーズで次のお題を使います。

> 「部長からの丸投げ：会議が多すぎる。AIで何とかして。来週までに運用案を出して」

この依頼文は意図的に不親切です。  
足りない情報を見つけ、質問を設計し、実行可能な設計図に落とし込みます。

## 必須成果物

このコースでは、以下の2ファイルを育てながら進めます。

1. `design-spec.md`（日本語の設計書本体）
2. `design-change-log.md`（仕様変更対応の履歴）

### `design-spec.md` の推奨構成

```md
# 設計書タイトル
## 1. 目的と成功条件
## 2. 前提・制約
## 3. 変数一覧（5W1H）
## 4. 処理フロー（手順）
## 5. 条件分岐（If-Then）
## 6. 例外時の対応
## 7. 未確定事項と確認質問
```

### `design-change-log.md` の推奨構成

```md
# 仕様変更履歴
| 日付 | 変更内容 | 影響範囲 | 差し替え方針 | 備考 |
|---|---|---|---|---|
```

## 評価軸（全フェーズ共通）

1. 問いの回数と質  
Why/Howの深掘りがあり、前提を検証できているか
2. 言語化の粒度  
第三者や別のAIが読んでも同じ結果を再現できるか
3. 失敗からのリカバリー  
AI回答のズレを他責にせず、条件定義の不足に戻って修正できるか

## 進行ルール

- 正解プロンプト集は配布しません
- 1回で正解を狙わず、質問を改善し続けます
- 迷ったら「前提」「制約」「成功条件」の3点に戻ります
- すべてのフェーズで提出課題があります（このガイド章を除く）

## このあと

次章から、4つのフェーズを順番に実施します。  
順序を飛ばさず、必ず `design-spec.md` と `design-change-log.md` を更新してください。
